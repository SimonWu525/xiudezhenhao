## 1. 项目概述

### 1.1 项目背景

面向用户提供一个低门槛、直接可用的老照片修复工具。用户上传原始照片后，系统基于整图完成图像增强处理，并输出可预览、可下载的结果图。

### 1.2 项目目标

建设一个单页完成的老照片处理工具，支持以下两项核心能力：

1. 一键修复  
   对老照片进行清晰度增强、划痕修复、噪点清理、细节还原。
2. 一键修复并上色  
   在修复基础上进行自然色彩还原，输出更接近真实观感的照片结果。

### 1.3 项目原则

- 用户上传后可直接处理，无需额外步骤
- 所有结果均基于用户原图生成
- 结果优先保证人物、服饰、姿态、构图一致性
- 页面操作简单明确，适合普通用户
- 失败场景必须清晰反馈，允许重试

## 3. 用户与场景

### 3.1 目标用户

- 有家庭旧照片修复需求的普通用户
- 希望将黑白老照片恢复为清晰或自然上色版本的用户
- 无专业修图经验，希望低成本快速获得结果的用户

### 3.2 核心使用场景

- 场景 A：修复老旧照片  
  用户上传一张模糊、泛黄、带折痕或噪点的老照片，点击“一键修复”，得到清晰增强版本。
- 场景 B：修复并上色  
  用户上传一张黑白或褪色照片，点击“一键修复并上色”，得到自然色彩还原版本。
- 场景 C：结果下载与分享  
  用户完成处理后，预览效果满意，下载高清结果图进行保存或分享。

## 4. 功能需求

### 4.1 图片上传

**功能说明**  
用户选择本地图片上传，系统完成格式与大小校验后展示原图预览。

**支持格式**

- JPG
- JPEG
- PNG
- WEBP

**上传限制**

- 单张图片大小：建议不超过 15MB
- 最低分辨率：不低于 512 × 512
- 最大边长：建议不超过 4096 px

**交互要求**

- 上传区域支持点击上传
- 上传成功后立即显示原图预览
- 上传失败时展示明确错误提示

**错误提示文案**

- 文件格式不支持，请上传 JPG、PNG 或 WEBP 图片
- 图片过大，请上传 15MB 以内的图片
- 图片尺寸过小，请上传更清晰的照片

### 4.2 一键修复

**功能说明**  
用户点击“一键修复”后，系统对整图执行老照片增强处理。

**处理目标**

- 去除明显噪点
- 修复轻中度划痕与折痕
- 增强清晰度
- 修复轻微模糊
- 保留原始人物特征、服饰、构图与时代感

**输出要求**

- 输出为单张结果图
- 默认输出格式：PNG
- 输出分辨率不低于输入主区域视觉质量

**按钮文案**  
一键修复

### 4.3 一键修复并上色

**功能说明**  
用户点击“一键修复并上色”后，系统对整图执行修复与自然上色处理。

**处理目标**

- 完成照片修复
- 恢复自然色彩
- 保持肤色、服饰、背景颜色整体协调
- 避免过饱和、过艳、过强滤镜感

**输出要求**

- 输出为单张结果图
- 默认输出格式：PNG
- 色彩风格以“自然还原”为默认策略

**按钮文案**  
一键修复并上色

### 4.4 处理中状态

**功能说明**  
用户触发处理任务后，页面进入处理中状态，避免重复点击。

**状态要求**

- 显示加载中动效
- 按钮置灰不可重复点击
- 页面展示当前任务状态文案

**建议文案**

- 正在修复照片，请稍候
- 正在修复并上色，请稍候

**交互要求**

- 处理完成前不可再次提交新任务
- 用户刷新页面后不保留处理中状态
- 同一时刻仅允许单图单任务运行

### 4.5 结果预览

**功能说明**  
处理完成后展示结果图，并支持与原图进行视觉对比。

**展示要求**

- 原图与结果图并排展示
- 支持切换查看“原图 / 修复结果”
- 支持放大查看结果图细节
- 支持重新上传新图片

**结果区按钮**

- 下载结果图
- 重新上传
- 再试一次（仅失败时显示）

### 4.6 下载结果图

**功能说明**  
用户可将处理后的图片下载到本地。

**要求**

- 文件名自动生成
- 命名规则统一
- 浏览器默认下载

**命名规则**

- 修复：`restore_<timestamp>.png`
- 修复并上色：`colorize_<timestamp>.png`

### 4.7 错误处理

**功能说明**  
后端处理失败时，前端应准确反馈原因，并提供重试入口。

**错误类型**

- 上传失败
- 模型处理失败
- 返回结果为空
- 图片解析失败
- 服务超时

**展示要求**

- 以页面内错误提示展示
- 错误提示不暴露技术细节堆栈
- 提供“再试一次”按钮

**建议文案**

- 当前处理失败，请稍后重试
- 图片解析失败，请更换图片后重试
- 服务繁忙，请稍后再试

## 5. 用户流程

### 5.1 主流程

1. 用户进入工具页面
2. 用户上传一张照片
3. 页面展示原图预览
4. 用户点击：
   - 一键修复
   - 或一键修复并上色
5. 页面进入处理中状态
6. 后端返回结果图
7. 页面展示结果预览
8. 用户下载结果图或重新上传下一张图片

### 5.2 异常流程

1. 用户上传图片
2. 用户触发处理
3. 后端返回失败
4. 页面展示错误提示
5. 用户点击“再试一次”或重新上传图片

## 6. 页面结构与交互

### 6.1 页面模块

单页结构分为 5 个模块：

1. 页面标题区
2. 上传区
3. 原图预览区
4. 操作按钮区
5. 结果展示区

### 6.2 页面状态定义

- 状态 A：初始状态
  - 显示上传入口
  - 不显示结果区
  - 操作按钮不可用
- 状态 B：已上传
  - 显示原图预览
  - 显示两个主操作按钮
  - 结果区为空
- 状态 C：处理中
  - 显示加载态
  - 所有主按钮禁用
  - 保留原图预览
- 状态 D：处理成功
  - 显示结果图
  - 显示下载按钮
  - 支持重新上传
- 状态 E：处理失败
  - 显示错误提示
  - 显示再试一次
  - 支持重新上传

## 8. 后端需求

### 8.1 服务职责

后端提供统一图片处理服务，负责：

- 接收上传图片
- 校验 MIME 类型与尺寸
- 转换为模型所需输入格式
- 调用图像处理模型
- 返回处理结果
- 记录日志与错误信息

### 8.2 技术职责拆分

建议后端拆分为以下模块：

1. 请求接入层
   - 参数校验
   - 文件校验
   - 请求 ID 生成
2. 图片预处理层
   - 读取图片
   - 标准化格式
   - 压缩与编码
   - 统一为模型输入格式
3. 模型调用层
   - 根据任务类型调用对应提示词模板
   - 执行图片处理请求
   - 解析模型返回图像结果
4. 结果输出层
   - 将结果存储为临时文件或对象存储
   - 返回可访问结果地址或 Base64
5. 日志监控层
   - 记录耗时、错误码、文件信息、任务类型

## 9. 接口定义

### 9.1 上传并处理接口

**接口地址**  
`POST /api/photo/restore`

**功能说明**  
接收前端上传的图片与任务类型，返回处理结果。

**Request**

- `Content-Type: multipart/form-data`

参数：

| 参数名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| file | File | 是 | 用户上传的图片 |
| taskType | string | 是 | 任务类型：restore / colorize |
| requestId | string | 否 | 前端生成的请求标识 |

`taskType` 枚举：

- `restore`：一键修复
- `colorize`：一键修复并上色

**Response - Success**

```json
{
  "success": true,
  "requestId": "req_20260311_001",
  "taskType": "restore",
  "resultImageUrl": "https://cdn.xxx.com/result/restore_20260311_001.png",
  "width": 2048,
  "height": 2048
}
```

**Response - Error**

```json
{
  "success": false,
  "requestId": "req_20260311_001",
  "errorCode": "MODEL_PROCESS_FAILED",
  "message": "当前处理失败，请稍后重试"
}
```

### 9.2 错误码定义

| 错误码 | 含义 | 前端文案 |
| --- | --- | --- |
| INVALID_FILE_TYPE | 文件格式错误 | 文件格式不支持，请重新上传 |
| FILE_TOO_LARGE | 文件过大 | 图片过大，请上传 15MB 以内图片 |
| IMAGE_DECODE_FAILED | 图片解析失败 | 图片解析失败，请更换图片 |
| MODEL_PROCESS_FAILED | 模型处理失败 | 当前处理失败，请稍后重试 |
| RESULT_EMPTY | 结果为空 | 当前处理失败，请稍后重试 |
| REQUEST_TIMEOUT | 请求超时 | 服务繁忙，请稍后再试 |

## 10. 模型执行规范

### 10.1 输入要求

模型调用必须包含：

- 用户上传原图
- 正确 MIME 类型
- 对应任务类型
- 标准化指令模板

### 10.2 指令模板

任务 A：一键修复

> Restore this old photo while preserving the original identity, facial structure, clothing, pose, composition, and historical atmosphere. Remove scratches, noise, blur, fading, and minor damage. Improve clarity naturally. Do not add new objects. Do not alter the subject.

任务 B：一键修复并上色

> Restore and colorize this old photo while preserving the original identity, facial structure, clothing, pose, composition, and historical atmosphere. Remove scratches, noise, blur, fading, and minor damage. Add natural and realistic colors. Do not add new objects. Do not alter the subject.

### 10.3 输出要求

- 必须返回单张图像结果
- 输出图片需可正常解码
- 输出内容与输入主体一致
- 不允许返回文本代替结果

## 11. 数据与存储

### 11.1 文件处理策略

- 原图仅用于本次任务处理
- 结果图临时存储
- 建议使用对象存储服务保存结果图，设置过期时间

### 11.2 存储建议

- 原图：内存临时处理或短时缓存
- 结果图：对象存储，保存 24 小时
- 日志：服务端持久化

### 11.3 日志字段

建议至少记录以下字段：

| 字段名 | 说明 |
| --- | --- |
| requestId | 请求唯一标识 |
| taskType | 任务类型 |
| fileName | 原始文件名 |
| fileSize | 文件大小 |
| mimeType | 文件类型 |
| width | 输入宽度 |
| height | 输入高度 |
| modelName | 使用模型名 |
| latencyMs | 处理耗时 |

## 13. 验收标准

### 13.1 功能验收

- 用户可成功上传支持格式图片

输出结果需满足：

- 主体人物身份一致
- 脸部结构不明显变形
- 构图不发生明显漂移
- 服饰与姿态保持一致
- 修复结果清晰度明显优于原图
- 上色结果自然，无明显夸张偏色

### 13.3 性能验收

- 单次请求平均响应时间：目标 8s 以内
- 峰值情况下前端不卡死
- 同一请求不出现重复返回

## 18. 最终执行结论

本期产品按单图上传、整图处理、两种任务、一页完成的方式落地执行。

上线最小可用版本必须具备：

- 上传
- 一键修复
- 一键修复并上色
- 结果预览
- 下载
- 错误提示

**研发执行口径**  
前端先做单页完整流程，后端先打通单接口模型链路，测试围绕“成功处理 + 明确失败 + 可下载结果”三件事验收。不要搞花活，先把这条主链打穿，产品就活了。
